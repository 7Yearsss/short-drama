# 内容运营发布稳定完整版设计

## 背景

当前后台已经支持管理员登录、剧集创建、封面上传、集数视频上传、转码队列、失败重试、剧集/集数上架，以及手动授予单剧解锁。下一阶段的目标不是扩展成完整 CMS，而是把内容运营中最核心、最高风险的发布链路做稳：一部剧可以安全地从草稿进入上架状态，上架后可以持续更新新集，也可以在传错视频时不影响线上观看地完成替换。

本设计采用“发布稳定完整版”范围：覆盖单剧运营工作台、发布检查、持续更新、免费剧逻辑、下架状态、视频替换、首页排序、轻量操作日志和后台搜索筛选；明确不做定时上架、批量上传、素材库、分类标签、草稿预览、推荐位/Banner。

## 目标

- 让管理员可以围绕一部剧完成完整运营：编辑资料、上传新集、上架/下架、替换视频、查看发布检查和操作日志。
- 让新集发布保持可控：上传后先转码，转码成功进入待上架，管理员确认后才公开。
- 让已发布集数可以安全替换视频：确认替换前，前台继续播放旧视频。
- 让免费剧语义明确：`unlockPriceCents = 0` 表示整部剧免费，所有已发布集数无需登录即可播放。
- 让首页展示可运营：支持简单排序，并让最近更新的连载剧自然前移。
- 让关键内容操作可追溯：记录轻量操作日志，便于排查误改、误上架、误替换。

## 非目标

- 不做定时上架或排期发布。
- 不做多文件批量上传；只优化连续单集上传体验。
- 不做素材库、历史封面管理、旧 R2 文件删除或图片裁剪。
- 不做分类、标签、专题、Banner 或推荐位。
- 不做草稿前台预览。
- 不做视频版本历史、回滚或新旧视频对比预览。
- 不做已发布内容硬删除；本阶段以“下架”为主。

## 后台信息架构

`/admin` 保持为全局内容列表页，职责是“找和排”：

- 搜索剧名。
- 按上架状态筛选：全部、草稿、已上架、已下架。
- 按更新状态筛选：全部、连载中、已完结、暂停更新。
- 展示总剧集、已上架、草稿/下架等运营概览。
- 展示每部剧的上架状态、更新状态、价格/免费标识、免费集数、最近更新时间、运营排序。
- 提供进入单剧工作台的入口。
- 支持编辑或调整 `sortOrder`，用于首页运营排序。

`/admin/series/:id` 从“集数管理页”升级为“单剧运营工作台”，职责是“改和发”：

- 基础信息区：标题、简介、封面、解锁价格、免费集数、更新状态、上架状态。
- 发布检查区：展示硬阻止项和软提醒项。
- 集数管理区：上传新集、查看转码状态、上架/下架集数、失败重试、替换视频。
- 前台入口：剧集已上架时显示“查看前台”；未上架时不做草稿预览。
- 最近操作日志区：显示该剧相关的关键后台操作。

## 数据模型

在现有 Prisma 模型上做增量扩展，不重做 CMS。

### Series

新增或扩展字段：

- `status`: 扩展为 `draft | published | offline`。
- `updateStatus`: `ongoing | completed | paused`，表示连载中、已完结、暂停更新。
- `sortOrder`: 数字，首页运营排序使用。
- `lastPublishedEpisodeAt`: 最近一次集数上架时间。
- `publishedAt`: 首次上架时间。
- `offlineAt`: 下架时间。

`unlockPriceCents = 0` 表示免费剧；`unlockPriceCents > 0` 表示付费剧。

### Episode

扩展字段：

- `status`: 扩展为 `processing | draft | published | failed | offline`。
- `publishedAt`: 集数首次上架时间。
- `offlineAt`: 集数下架时间。
- `replacementR2Key`: 待确认替换视频的 R2 key。
- `replacementDurationSeconds`: 待确认替换视频时长。
- `replacementStatus`: `processing | ready | failed`，为空表示没有替换任务。
- `replacementUploadError`: 替换转码/上传失败信息。
- `replacementTempVideoPath`: 替换失败后可重试时保留的临时原始文件路径。

第一版不做 Episode 版本表。替换视频只需要一个待确认暂存区：确认后把 replacement 字段写入正式 `r2Key` / `durationSeconds`，再清空 replacement 字段。

### AdminAuditLog

新增模型：

- `id`
- `adminId`
- `action`
- `targetType`
- `targetId`
- `seriesId`
- `metadata` JSON
- `createdAt`

`seriesId` 用于单剧工作台快速查询最近操作，不依赖从 `metadata` 里解析。

## 内容状态

剧集状态：

- `draft`: 草稿，从未正式上架。
- `published`: 已上架，前台可见。
- `offline`: 已下架，前台不可见，后台保留记录。

集数状态：

- `processing`: 视频上传成功，正在转码/上传 R2。
- `draft`: 转码成功，待管理员上架。
- `published`: 已上架，前台可见。
- `failed`: 转码/上传失败，可重试或重新上传。
- `offline`: 已下架，前台不可见，后台保留记录。

更新状态：

- `ongoing`: 连载中。
- `completed`: 已完结。
- `paused`: 暂停更新。

## 发布流程

### 剧集上架

管理员在单剧工作台点击“上架剧集”时，系统先运行发布检查：

- 有硬阻止项：不允许上架，要求修正。
- 只有软提醒项：展示确认提示，管理员确认后继续。
- 无问题：直接上架。

上架成功后：

- `Series.status = published`
- 如果是首次上架，写入 `publishedAt`
- 记录 `series.publish` 操作日志

### 新集上传与续更

新集上传保留手动确认发布：

1. 单剧工作台根据现有最大集数自动建议下一集编号。
2. 管理员可以手动修改编号。
3. 编号重复时直接禁止提交。
4. 编号跳号时弹确认，例如已有 1-10 集，管理员提交第 12 集。
5. 上传后创建 `Episode.status = processing`。
6. 转码和 R2 上传成功后进入 `Episode.status = draft`，表示待上架。
7. 管理员点“上架”后进入 `Episode.status = published`。
8. 写入 `Episode.publishedAt`，并更新所属剧 `lastPublishedEpisodeAt`。
9. 记录 `episode.upload` 和 `episode.publish` 操作日志。

上传成功后表单应清空视频文件，集数自动建议为下一集，列表继续自动刷新 `processing` 状态，便于连续单集上传。

### 下架

已发布内容以“下架”为主：

- 下架剧集：`Series.status = offline`，写入 `offlineAt`，前台不再展示该剧。
- 下架集数：`Episode.status = offline`，写入 `offlineAt`，前台不再展示该集。
- 下架需要二次确认并记录操作日志。
- 本阶段不开放已发布剧集/集数硬删除。

## 视频替换流程

已发布集数支持安全替换视频，但确认前不影响线上旧视频。

流程：

1. 已发布集数显示“替换视频”操作。
2. 管理员上传替换文件。
3. 系统不修改当前 `r2Key`，前台继续播放旧视频。
4. 替换任务进入 `replacementStatus = processing`。
5. 转码和 R2 上传成功后：
   - 写入 `replacementR2Key`
   - 写入 `replacementDurationSeconds`
   - `replacementStatus = ready`
6. 工作台展示三个操作：
   - 预览新视频
   - 确认替换
   - 放弃替换
7. 预览新视频使用 replacement 文件签名播放地址；不强制预览后才能确认。
8. 确认替换后：
   - `r2Key = replacementR2Key`
   - `durationSeconds = replacementDurationSeconds`
   - 清空 replacement 字段
   - 前台开始播放新视频
   - 记录 `episode.replacement_confirm` 操作日志
9. 放弃替换时清空 replacement 字段，保留线上旧视频。
10. 替换失败时：
   - `replacementStatus = failed`
   - 写入 `replacementUploadError`
   - 不影响旧视频播放
   - 允许重新上传或在保留临时文件时重试

第一版不做版本历史、回滚、新旧视频对比预览或旧 R2 文件删除。

## 播放权限

播放权限优先级调整为：

1. 如果 `Series.unlockPriceCents === 0`，该剧是免费剧，所有已发布集数无需登录即可播放。
2. 否则，如果 `Episode.episodeNumber <= Series.freeEpisodeCount`，该集是试看集数，无需登录即可播放。
3. 否则需要登录，并且满足以下任一条件：
   - 用户有未过期会员。
   - 用户拥有该剧的单剧解锁记录。

免费剧仍保留 `freeEpisodeCount` 字段，但权限逻辑不使用它。后台在价格为 0 时弱化或提示“免费剧不使用试看集数”。

## 前台展示

首页剧集列表：

- 只展示 `Series.status = published` 的剧。
- 排序规则：
  1. `sortOrder desc`
  2. `lastPublishedEpisodeAt desc`
  3. `createdAt desc`
- 免费剧显示“免费观看”。
- 付费剧显示单剧解锁价、免费试看集数或等效文案。
- 展示更新状态：连载中、已完结、暂停更新。

剧集详情页：

- 只展示 `Episode.status = published` 且有正式 `r2Key` 的集数。
- 免费剧不显示锁定状态。
- 付费剧继续按试看集数、会员和单剧解锁显示锁定状态。
- 已下架集数不展示。

## 发布检查清单

硬阻止项：

- 没有剧名。
- 没有封面。
- 没有任何可发布或已发布集数。
- 首次上架时有集数正在转码，且没有任何已发布集数。
- 付费剧的 `freeEpisodeCount < 0`。

软提醒项：

- 没有简介。
- 存在失败集数。
- 集数不连续。
- 付费剧的免费集数大于当前已发布集数。
- 更新状态为已完结，但集数数量很少。

危险编辑需要二次确认：

- 价格改为 0：整部剧变为免费。
- 价格从 0 改为大于 0：后续集数可能变为需解锁。
- 调低免费集数：部分原本免费的集数可能变为锁定。
- 下架剧集或集数。
- 确认替换已发布视频。

## 操作日志

第一版只记录会影响前台展示、收费或观看的关键动作：

- 创建剧集。
- 编辑剧集基础信息。
- 上架/下架剧集。
- 上传新集。
- 上架/下架集数。
- 替换视频开始、确认替换、放弃替换、替换失败。
- 修改免费集数。
- 修改解锁价格，尤其是改成 0 或从 0 改成付费。
- 修改更新状态。
- 调整首页排序。

不记录：

- 管理员登录。
- 页面浏览。
- 普通列表查询。
- 上传表单打开/关闭。
- 播放预览。

## API 影响

需要新增或调整的接口方向：

- 管理员剧集列表支持搜索、筛选、排序。
- 管理员更新剧集基础信息时记录关键操作日志。
- 管理员上架剧集前可获取发布检查结果。
- 管理员上架/下架剧集。
- 管理员上架/下架集数。
- 管理员上传新集继续使用现有上传接口，但补充下一集建议、跳号确认在前端处理，后端继续强校验重复编号。
- 管理员上传替换视频。
- 管理员获取替换视频预览签名 URL。
- 管理员确认替换视频。
- 管理员放弃替换视频。
- 管理员查看单剧最近操作日志。
- 公开播放权限逻辑支持免费剧。
- 公开剧集列表和详情展示更新状态、免费剧状态和排序。

具体路径可以在实现计划中按现有路由风格细化，优先复用 `series.ts`、`episodes.ts`、`playback.ts`、上传队列和 R2 签名工具。

## 测试策略

后端优先测试：

- 免费剧所有已发布集数无需登录可播放。
- 付费剧仍按试看集数、会员、单剧解锁判断。
- 剧集上架检查：硬阻止和软提醒结果正确。
- 新集上传成功后进入 `draft`，不会自动公开。
- 集数上架会更新 `lastPublishedEpisodeAt`。
- `offline` 剧集和集数不会出现在公开接口。
- 已发布集数替换视频确认前旧 `r2Key` 不变。
- 替换确认后正式 `r2Key` 更新，replacement 字段清空。
- 替换失败不影响旧视频播放。
- 操作日志在关键动作后创建。

前端可用现有轻量测试方式补充：

- `/admin` 有搜索/筛选/排序控件。
- `/admin/series/:id` 呈现基础信息、发布检查、上传新集、替换视频和日志区域。
- 免费剧显示免费观看文案，不显示误导性的单剧解锁价。
- 已上架剧集显示查看前台入口。

## 待实现时注意

- 保持小步提交；后端改动先写测试再实现。
- 不要把 `offline` 误当成 `draft`，两者业务含义不同。
- 不要在替换视频上传时覆盖正式 `r2Key`。
- 不要引入定时发布、批量上传、素材库或标签系统。
- 注意 Next.js 可能自动修改 `apps/web/tsconfig.json`，提交前避免污染 diff。
